Blockchain-Based Music Platform Development

BY : Artem Dumchev

Contents
Week 1 Choosing a technology Stack.

The Foundation: Choosing A Tech Stack

In the first week of my journey to build a blockchain-based music platform, I've been diving deep into the critical task of choosing the right technology stack. It's a decision that feels as foundational as laying down the first brick of a house – everything else will be built upon it. So, here's how I've been approaching this crucial phase.

Frontend Decisions

I've already settled on React JS for the web interface as I have experience using it. Its component-based nature is a perfect fit for the dynamic and engaging UI I envision. React's vast ecosystem and community support mean I'm never too far from finding a solution or a new idea to implement.

Mobile App Development Decision

Choosing the right framework for the mobile app was a challenging decision – Flutter or React Native? Why React Native?

Development Speed: React Native allows for a shared codebase with the web platform, streamlining the development process, similarly to Flutter.

Performance: Flutter compiles to native code, which can lead to better performance in some scenarios. This performance edge can be particularly important for apps with complex animations or where device hardware access is critical.

Community: React Native (2015), being older, has a larger and more established community compared to Flutter (2017), meaning that I will be more likely to find solutions to the problems I will come across.

Previous Experience: I have used React JS before, which shares the same concepts of components, state management, and props that are fundamental to building dynamic user interfaces.

Despite Flutter's performance advantages and growing popularity, I've chosen React Native for its extensive community, familiarity and a smoother learning curve.

After deciding that I will use React Native. I completed a Codecademy Course.

Building the Backend

The backend part of the project is what I consider to be the most challenging. It will be responsible for complicated tasks such as NFT creation and Marketplace operations, uploading and streaming tracks.

User Authentication: A Dual Approach

Firebase Authentication: This has been my go-to for managing sign-ins via Gmail, thanks to its ease of setup and broad integration with various login methods. It provides a quick and secure way to handle user authentication.

Web3 Libraries: For users who want access to the crypto part of the app, I plan to use libraries like Web3.js and ethers.js to bridge their Ethereum wallets with our platform. This integration is vital for linking activities like NFT transactions and minting directly to their blockchain wallet.

Music Streaming and User Engagement

Firebase's Firestore: I'm leveraging Firestore for its real-time data handling capabilities, utilizing it to store user data, playlists, and song metadata. This ensures the platform remains dynamic and responsive.

Firebase Cloud Storage: This will act as the music file repository, providing the scalability and reliability needed for streaming. It's crucial to ensure all hosted music is legally uploaded by its creators or those with who have the rights.

Custom Backend APIs: Developed with Node.js and Express.js, these APIs will facilitate various user actions like curating playlists and marking favorites, bridging the database and user interface.

The NFT Marketplace - Creativity Plus Commerce

Smart Contracts: Utilizing Solidity, I'm setting up the base for the NFT transactions, which will handle the minting, sale, and transfer processes on the Ethereum blockchain.

IPFS: For decentralized storage of the actual music files linked to NFTs, IPFS presents a fitting solution that aligns with the blockchain's decentralized nature.

Web3.js and ethers.js Integration: This integration will allow the platform to interact with the smart contracts, enabling functionalities from NFT minting to marketplace transactions.

Marketplace System: A crucial component of the platform, this system will enable users to list, sell, and buy NFTs, fostering a community where creative value is acknowledged and rewarded.

Database Design

User Database

Purpose: Stores all user-related data, including authentication details, user profiles, preferences, and activity logs.

Contents: User IDs, email addresses, usernames, profile information. References or IDs to favorite songs, owned NFTs. Transaction history within the marketplace.

Functionality: Supports user authentication, profile customization, and personalization of the music listening experience.

Songs Database

Purpose: Contains all information about the songs available on the platform, including metadata and streaming data.

Contents: Song IDs, titles, artists, album information, genres. File URLs (for streaming), and any associated NFT data.

Functionality: Facilitates music streaming, browsing, and searching within the app.

Marketplace Database

Purpose: Manages all data related to the buying, selling, and trading of music NFTs on the platform's built-in marketplace.

Contents: Listings of NFTs for sale, including references to the corresponding song.Ownership records, transaction histories, and pricing information.

Functionality: Enables the marketplace operations, including listing NFTs for sale, executing transactions, and tracking ownership changes.

Note

It is useful to outline that the NFT will act more in the form of a donation rather than a proof of ownership. They will be associated with a track but that doesn't mean the person who owns the NFT owns the track. The artist retains full ownership of the track and the right to upload the track to any other platforms.

Development Model

Approach: Given the ambitious scope and the tight timeline, I've opted for an Agile development model. This iterative process with 2-week sprints allows for rapid adaptation, effective feature prioritization, and the incorporation of feedback at every stage.

Weekly Focus: This week has been foundational - selecting the tech stack and planning the development approach. Challenges are inevitable, but each decision is a step toward a platform that connects us with niche music genres and supports artists in continuing to create meaningful, culture-reflecting music.
